UI Overhaul v2

Six changes to the task manager, grouped into five phases ordered by risk and dependency. Every phase ends with a Codex E2E loop. This plan assumes FIX_PLAN.md ships first - drag-and-drop and the new task-detail sections build directly on the fixed update/subtask actions.

Mocks below are rebuilt from the real components (task-card.tsx, page.tsx, task-form.tsx) - annotate anything that doesn't match what you had in mind. Live demos are concept studies: timing and detail get tuned during implementation.

Where do finished subtasks go, exactly?

A · Computed view recommended
The Work log tab renders a merged, time-sorted feed of log entries + subtasks with status Finished. No schema change, fully reversible, un-finishing a subtask just moves it back.
B · Materialize a log entry
Finishing a subtask writes a real TaskLog row (needs a nullable subtaskId link + migration, plus rules for un-finish/delete). Heavier, only worth it if finished subtasks should carry logged minutes into the totals.
  • Tab state in the URL (?tab=log) so tabs deep-link and survive revalidation - not useState.
  • The three sections reuse the board's column styling at smaller scale; status changes still available via the select (keyboard path), drag-and-drop arrives in Phase 4.
  • The "Finished" option stays in the subtask status enum - the sections just don't render a Finished lane.
  • Depends on FIX_PLAN task 4 (awaited subtask actions with error states) - don't build tabs on fire-and-forget calls.

✔ Decided · Option A, computed view

The Work log tab renders a merged, time-sorted feed of log entries + finished subtasks. No schema change; un-finishing a subtask simply moves it back to its section.

Phase 3 · E2E loop (Codex)

  • Detail page defaults to Tasks tab; ?tab=log deep-links to Work log; browser back toggles tabs.
  • Create subtasks in each status → each renders in its section; statuses with no subtasks show their empty state.
  • Finish a subtask → it leaves the sections, appears in Work log per chosen semantics, tab badge count updates; un-finish → it returns.
  • Adding a plain work-log entry still works from the Work log tab; time totals stay correct.
  • Two-tab failure path (regression of FIX_PLAN task 4): delete a subtask in tab A, change its status in tab B → visible error, no unhandled rejection.

Demo uses raw HTML5 DnD - fine on desktop, dead on touch. That gap is exactly why the library choice below matters.

Library

@dnd-kit recommended
React-native API, built-in keyboard sensor (a11y for free), touch support, small core. The default choice for React kanbans; plenty of examples to learn from.
Atlassian pragmatic-drag-and-drop
Framework-agnostic, very performant, actively maintained (powers Trello/Jira). More wiring by hand in React; keyboard support is DIY.
Native HTML5 DnD
Zero dependencies, but no touch events, janky drag previews, and full custom a11y work. Cheapest to start, most expensive to finish.
  • On drop: useOptimistic moves the card immediately → await updateTask/updateSubtaskStatus → revalidation confirms. On failure: card snaps back + error toast.
  • Keyboard path stays: the status select (subtasks) / edit dialog (tasks) remain, and dnd-kit's keyboard sensor adds move-by-arrow-keys.
  • Decided: manual reordering within a column is in scope. Add a sortOrder Int column to Task and Subtask (migration backfills from the current deadline/created sort). Columns then order by sortOrder; drop position computes the new rank (integer gaps, reindex the column when gaps run out). Note the trade-off this buys: the board stops auto-sorting by deadline - overdue stays visible via the red highlight instead.
  • Board columns and Phase 3 sections share one <StatusDropZone> wrapper so the behavior is implemented once.

✔ Decided · @dnd-kit, with manual in-column reordering

@dnd-kit for cross-column status drags and within-column reordering, backed by the new sortOrder column described above.

Phase 4 · E2E loop (Codex)

  • Drag board card Inbox → Review: card moves instantly, column counts update, hard reload proves the status persisted.
  • Reorder two cards within a column → order survives a hard reload; creating a new task appends at a consistent position; reordering in one column doesn't shuffle other columns.
  • Drop back onto the source column: no-op, no server call fired (assert via network log).
  • Failure path: kill the DB (or stub the action to throw) mid-drag → card snaps back to origin + visible error, board still matches DB after reload.
  • Subtask drag across Phase 3 sections persists; dragging to "finished" (if exposed as a drop target) follows Q2 semantics.
  • Touch: drag works in a mobile-emulated viewport. Keyboard: pick up/move/drop a card without a mouse. prefers-reduced-motion: no lift/spring animations.
  • Regression: filters + archived states still render correctly after drags; archived cards are not draggable.

Phase 5 · 8-bit create animation low risk

On a successful Add task: the dialog crumbles into pixels, the pixels swirl into a point, a white 8-bit star forms, blinks, and vanishes. Pure CSS transforms + a few dozen generated pixel divs - no canvas, no library.

Chunkiness comes from steps() easing - movement snaps between frames instead of gliding, which is what sells the 8-bit feel.

  • Trigger correctness: animation plays only after createTask resolves - never on validation failure (dialog stays open with errors, exactly as now).
  • Implementation: on success, measure the dialog, overlay a fixed-position pixel layer (12-16px cells sampled from the dialog's palette), hide the dialog instantly, animate the layer. The board updates behind it via the normal revalidation while the star plays - no blocking.
  • Star rendered as one div with box-shadow pixel art (white + 1px ink outline over a briefly dimmed backdrop so white reads).
  • prefers-reduced-motion: skip straight to close. Escape hatch: any click skips the animation.
  • Micro-motion (same system, tiny scale): archive = card does a 2-frame pixel "drop", delete confirm = card crumbles down, drag pickup = 1px jitter. Optional - tick below.
  • Optional flavor: Press Start 2P for tiny accents only (like the "TASK EDITOR" kicker in the demo) - it's illegible at body sizes.

✔ Decided · Full scope

All three: the create-task star animation, archive/delete/drag micro-animations, and Press Start 2P pixel-font accents (kickers and column counts only - never body text).

Phase 5 · E2E loop (Codex)

  • Create a valid task → full sequence plays once, dialog is closed after, new card is on the board; record a GIF for review.
  • Submit an invalid task (server-rejected via direct estimate tampering) → no animation, dialog stays open with the error.
  • Emulate prefers-reduced-motion: reduce → instant close, no pixels, task still created.
  • Click mid-animation → skips cleanly. Create 3 tasks rapidly → no orphaned pixel layers left in the DOM.
  • No layout shift (board doesn't jump when the overlay mounts) and no console errors; animation total stays ≤ 1.5s.

The Codex loop (applies to every phase)

Per phase: implement → pnpm typecheck && pnpm check → run that phase's E2E list in a real browser (assert DB state via Prisma script, never eyeball-only) → any failure: fix and restart the loop → all green: commit feat: phase N - ..., then run the cumulative regression: all previous phases' E2E lists plus FIX_PLAN's final regression loop (fresh-DB provision, full journey, hostile pass, two-timezone pass). Phases land as separate commits so any one can be reverted alone.

Risks worth naming

DnD × revalidation jank. revalidatePath re-renders the board after every drop; without useOptimistic the card teleports back for a frame. Optimistic state is a hard requirement of Phase 4, not a nice-to-have.
Animation vs server truth. Playing the star before createTask resolves means animating a task that might not exist. Trigger strictly on resolve; the E2E invalid-submit test guards this.
Manual ordering changes board semantics. With sortOrder as the column sort, the board no longer auto-floats the nearest deadline to the top - overdue signaling relies on the red highlight. If that feels wrong in practice, a "sort by deadline" toggle is a cheap follow-up.
Scope note: the detail-page sections are Inbox / Review / Ongoing; the board keeps its 4 columns including Finished (no annotation said otherwise). If Finished should leave the board too, that's a new decision.
