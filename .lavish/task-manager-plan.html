Exact Prisma schema additions (click to expand)
model Client {
  id    Int    @id @default(autoincrement())
  name  String @unique
  tasks Task[]
}
model Label {
  id    Int    @id @default(autoincrement())
  name  String @unique
  color String // tailwind-safe hex, rendered as badge
  tasks Task[]
}
model Task {
  // ...existing fields...
  deadline           DateTime?
  estimateMinMinutes Int?      // "2 to 5 hours" = 120
  estimateMaxMinutes Int?      // = 300, zod-capped at 300
  archivedAt         DateTime? // null = active
  clientId           Int?
  client             Client?   @relation(fields: [clientId], references: [id])
  labelId            Int?
  label              Label?    @relation(fields: [labelId], references: [id])
  subtasks           Subtask[]
  logs               TaskLog[]
}
model Subtask {
  id               Int        @id @default(autoincrement())
  taskId           Int
  task             Task       @relation(fields: [taskId], references: [id], onDelete: Cascade)
  title            String
  status           TaskStatus @default(Inbox)
  estimatedMinutes Int?       // target 30-45
  createdAt        DateTime   @default(now())
}
model TaskLog {
  id           Int      @id @default(autoincrement())
  taskId       Int
  task         Task     @relation(fields: [taskId], references: [id], onDelete: Cascade)
  note         String
  minutesSpent Int?     // optional: compare actual vs estimate
  createdAt    DateTime @default(now())
}

Notes: onDelete: Cascade means deleting a task removes its subtasks and logs (the delete confirm dialog states this). archivedAt keeps the last status intact, so unarchiving restores the card to its column. Estimates stored as integer minutes - no float math, easy to sum.

The 5-hour split rule in the form

Estimate is entered as a min-max range in hours. Zod caps the max at 300 minutes server-side; the form blocks submission above 5h and tells you to split, nudging from 3h up.

Conventions carried through every phase

  • Server actions only (no API routes) - matches the existing createTask pattern: Zod parse → Prisma → revalidatePath.
  • All ids validated with z.coerce.number().int().positive(); mutations verify the row exists before acting.
  • Board filtering happens in the Prisma query via searchParams, not client-side - stays a server component, URLs are shareable.
  • No new dependencies. Native <dialog>, native <input type="date">, Tailwind only.
  • Verify with pnpm typecheck + pnpm check after each phase (dev server assumed running).

6 · Security audit

  • Injection (A03) - all DB access through Prisma parameterized queries; every server action input parsed with Zod (ids coerced to positive ints, strings length-capped) before touching the DB.
  • XSS (A03) - all user content (titles, descriptions, log notes, label names) rendered through React's default escaping; no dangerouslySetInnerHTML anywhere in the plan.
  • Broken access control (A01) - the app has no auth: anyone who can reach the server can create/delete anything via server actions. Acceptable for a personal localhost tool; must add auth (Clerk fits your stack) before deploying anywhere public. Flagging, not building it in this plan.
  • Insecure design / mass assignment (A04) - actions whitelist fields via explicit Zod schemas; raw FormData is never spread into Prisma calls. Destructive ops (delete) require explicit confirm and validate the id server-side regardless of UI state.
  • Unvalidated redirects / IDOR-ish ids - /tasks/[id] coerces the param and returns notFound() for missing rows instead of leaking errors; label/client filter params validated before hitting the query.
  • DoS via unbounded input (A05) - length caps on every string (title 200, description/note 2000, label/client name 50); estimate ints range-checked (0 < x ≤ 300).

7 · Risks & edge cases considered

Migration on existing data

All new Task columns are nullable, so the migration is non-destructive to current tasks. Dropping the unused Post model is the only destructive step - it holds no real data.

Deleting a label/client in use

Not exposed in the UI this round. Relations are optional, so if added later it's SetNull, never cascade through tasks.

Archive vs Finished confusion

Finished = done, still visible on the board. Archived = hidden everywhere unless "show archived" is on. Archiving preserves status for clean restore.

Card getting crowded

Card shows at most: title, label badge, client, deadline, estimate, subtask count, 3 small buttons. Everything else lives on the detail page.
